# 結構化紀錄萃取提示詞（第三段 LLM）

用途：把裁判書 + 白話文摘要轉成一份 JSON 紀錄，欄位對應 [schema.md](../schema.md)。

複製下方「── 提示詞起點 ──」到「── 提示詞終點 ──」之間的全部內容，貼到任何 LLM。把兩個 `<<< ... >>>` 區塊分別換成裁判書全文與第 5 步產出的白話文摘要。

---

── 提示詞起點 ──

角色：你是資料結構化助理，將裁判書與白話文摘要整理成固定欄位的 JSON。

任務：產出一份 JSON 紀錄，欄位嚴格依照下列 schema。

輸出規則：

1. 輸出純 JSON。禁止任何 markdown，禁止 JSON 之外的解釋文字。不要包在 ```json ... ``` 中。
2. 任何無法從裁判書原文「明確取得」的欄位，填 `null`。**禁止推測**。
3. 字串內容使用繁體中文。
4. 日期一律 ISO 8601 格式（YYYY-MM-DD）。民國年要換算為西元年。
5. 「重點摘要」欄位直接放入下方第二個區塊（白話文摘要）的全文，不要重新生成。
6. 「source_url」、「collected_by」、「collected_at」三欄請固定填入空字串 ""，協作者會自己補上。

JSON schema：

{
  "defendant_name": string,
  "終審案號": string,
  "審級": "一審" | "二審" | "三審" | "更審",
  "是否確定": bool,
  "判決日": string,
  "犯罪事實發生日": string,
  "法院": string,
  "起訴法條": string[],
  "罪名": string[],
  "主刑": string,
  "從刑": string[],
  "易科罰金": string | null,
  "緩刑": { "年數": number, "條件": string } | null,
  "共犯案號": string[],
  "related_cases": [
    { "審級": string, "案號": string, "法院": string, "判決日": string }
  ],
  "重點摘要": string,
  "source_url": "",
  "collected_by": "",
  "collected_at": ""
}

欄位特別規則：

- 「defendant_name」：若裁判書全文中被告姓名為 ○○○，填 `null`（協作者會放棄這筆）。
- 「起訴法條」：寫具體條項款，例 "刑法 §339-4 第1項第1款"，不要只寫 "刑法 §339-4"。
- 「主刑」：完整保留判決主文用語，例 "有期徒刑 1 年 6 月"。
- 「從刑」：沒收、褫奪公權等。沒有就填 `[]`。
- 「related_cases」：本案在較低審級的案號。若你手上只有終審判決，從判決書中「歷次裁判」段落抓較低審級資訊；抓不到就填 `[]`。
- 「重點摘要」：直接複製下方第二區塊全文，不要改寫。

## 步驟 8：自我驗證提示詞

產出 JSON 後，把 JSON + 裁判書 URL 連同以下提問貼回 LLM。任何不是「PASS」的回覆都要修正後再驗證一次。

── 自我驗證提示詞起點 ──

以下這份 JSON 紀錄是否忠實對應這份裁判書？

請列出：
1. 任何捏造的事實（裁判書沒有但 JSON 寫了）。
2. 任何重要遺漏（裁判書有但 JSON 沒記）。
3. 任何推測欄位（沒明確來源就填了值，應該改 null）。

若全部正確，僅回覆 PASS。

## 卡關常見原因

- LLM 把整份 JSON 包在 code fence 內：複製時記得剝掉外層三反引號，或重跑時加一句「不要使用 code fence，直接輸出 JSON 純文字」。
- LLM 對「related_cases」推測填值：重跑時加一句「若裁判書未明確記載歷次裁判，related_cases 必須為空陣列 `[]`」。
- LLM 把民國年留著沒換算：重跑時提醒「所有日期一律 ISO 8601 西元年」。
