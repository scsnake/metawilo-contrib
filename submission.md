# 提交方式

## 首選：GitHub Pull Request

若站長有 GitHub repo（請先確認；若尚無，協助站長開一個），按以下方式提交：

1. Fork 站長的 repo → 在自己 fork 上建立分支，例：`add-records-王小明-batch1`。
2. 新增檔案到 `records/<court>_<終審案號>.json`。
3. **一份 JSON 一筆紀錄**，不要把多筆合成一個檔。
4. PR 描述使用以下模板：

```
## 鎖定目標
（一句話描述：依姓名 / 依罪名 / 依公告）

## 本批數量
N 筆

## 自我檢查清單
- [ ] 全部紀錄為確定判決（是否確定 = true）
- [ ] 全部紀錄為成年被告
- [ ] 全部紀錄已通過 prompts/classify.md 分類為 eligible
- [ ] 已對照 exclusions.md，無命中排除條件
- [ ] 重點摘要為純文字，無 *、#、>、- 等 markdown 符號
- [ ] 每筆都已用 LLM 自我驗證（workflow.md 第 8 步）
- [ ] source_url 均為 judgment.judicial.gov.tw 連結

## 鑑定者
（你的代號 / GitHub handle）

## 備註
（任何站長需要知道的事，例如「本批含 1 筆更生人疑慮，請站長判斷」）
```

5. 站長 review 後合併。**站長保留拒絕、修改、移除任何紀錄的權利**。

---

## 備用方式

若站長偏好其他通道：

- **電子郵件**：把所有 JSON 內容直接貼在信件中（不要當附件，方便站長 grep）。
- **共享雲端資料夾**：Google Drive / Dropbox，按 `records/` 結構放。

**請避免**：

| 格式 | 為什麼不建議 |
|---|---|
| CSV | 重點摘要含換行會破壞欄位。 |
| Google Forms | 結構化欄位會被攤平、難維護。 |
| Markdown 表格 | 不利規模化，超過幾十筆就難讀。 |
| 純文字 chat 訊息 | 無法回溯，站長無法批次處理。 |

---

## 站長收件後的處理（建議流程）

1. 抽樣對照 `source_url` 與原裁判書，確認摘要無捏造。
2. 必要時要求修正。
3. 合併進 `records/` 目錄，並同步到網站。
4. 對於日後被法院更正、上級審推翻、或當事人合理申訴的紀錄，**承諾在合理時間內移除或加註**。

---

## 站長承諾事項（建議公開於網站）

- 僅刊登**確定判決有罪**的成年被告紀錄、公告通緝犯、其他公開公告。
- 收到合理更正、下架請求時，在合理時間內處理。
- 不販售資料、不轉授權給第三方。
- 揭露目的限於公共利益警示，不作其他用途。

---

## 更正與下架政策（提交時即視為同意）

協作者提交時即視為同意：站長可在合理請求下移除或修改紀錄，**不需事先通知協作者**。

若你發現自己之前提交的某筆紀錄有錯（例如後來確定無罪、發回更審），請立即發 issue 通報站長。
